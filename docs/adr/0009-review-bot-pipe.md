## Context

On a Gerrit `patchset-created` event the review-bot receiver must review the change
and cast the deterministic `LLM-Review` vote (ADR-0013: a single label gated by a
submit requirement). For d251 we prove the PIPE with a real-but-minimal review; the
full four-pass kernel reviewer and the real dogfood are **b744-WS6**, which must plug
into the SAME seam without reshaping the receiver. The receiver IMPORTS the review
code as a library â€” it is NOT the stdio MCP (`rebar-mcp`) over HTTP, because Gerrit
speaks plain webhooks, not MCP JSON-RPC (ADR-0007).

The gate is **fail-closed**: submit REQUIRES the vote, so a missed event, a dropped
vote, or a broken receiver only ever DELAYS review (the change stays unsubmittable) â€”
it can never let an unreviewed change merge.

## Decision

### 1. The formal verdictâ†’label adapter (`code_review_decision`)

A single function is the contract between "what the LLM thinks" and "what value the
label gets":

```
code_review_decision(diff_text: str, repo_root, ref: str) -> {decision, message, findings}
```

`decision` is `"PASS"` or `"BLOCK"`. For the d251 proven pipe it is implemented over
the EXISTING single-pass `rebar.llm.review_code(...)`:

- It calls `review_code(diff_text=diff_text, source="local", repo_root=repo_root, ref=ref)`.
  **`source="local"`** because the receiver has ALREADY cloned the change ref into
  `repo_root`; `attested` mode would git-fetch `origin` and review the wrong state (a
  not-yet-merged patchset's ref lives only in the clone, not on origin).
- **Severity-threshold mapping.** `review_code` returns a `review_result` whose
  `findings[]` each carry a `severity` from the closed vocabulary
  `{critical, high, medium, low, info}` (`rebar.llm.findings.SEVERITIES`). The rule:
  **BLOCK if any finding's severity is in the configured blocking set (default
  `{critical, high}`), else PASS.** The blocking set is config-driven
  (`BLOCKING_SEVERITIES`), not hardcoded, because b744-WS6 changes the scoring logic.

**b744-WS6 reimplements this same signature** over
`gate_dispatch.produce_code_review_verdict` (a typed `code_review_verdict
{verdict: PASS|BLOCK, â€¦}`) â€” drop-in, no receiver change. The function is kept small
and free of receiver/Gerrit concerns precisely so that swap is mechanical.

### 2. Decisionâ†’integer mapping (config-driven, fail-closed)

| Code-review decision         | `LLM-Review` value          | Effect                     |
|------------------------------|-----------------------------|----------------------------|
| `PASS` (no blocking findings)| `LLM_REVIEW_MAX_VALUE` (+1) | change becomes submittable |
| `BLOCK` (blocking findings)  | `LLM_REVIEW_BLOCK_VALUE` (-1)| change stays unsubmittable |
| error / no / unparseable result | `LLM_REVIEW_BLOCK_VALUE` (-1)| fail-closed: stays unsubmittable |

The MAX/BLOCK values (and the label range, owned by d251 `project.config`) are
env/SSM-sourced (`config.py`), not hardcoded, since b744-WS6 changes the scoring.
A **MAX is cast ONLY on an explicit PASS** â€” every error path (adapter exception, a
missing/bad result, a clone/diff failure, a vote-POST failure) leaves the change
unsubmittable. No half-cast MAX is ever possible on failure.

### 3. Outbound vote shape

`POST /a/changes/{id}/revisions/{rev}/review` with `tag:"autogenerated:rebar"`,
`notify:"NONE"`, `labels:{"LLM-Review": <value>}`, and a `robot_comments` payload.
For the minimal shim the reviewer may produce no file-anchored findings, so the
receiver posts ONE patchset-level robot comment under the magic path
`"/PATCHSET_LEVEL"` (a robot comment with NO `path` is a Gerrit 400). The
`robot_run_id` is **deterministic** (`<change_id>-<revision>`) so a webhook and a
backfill for the same patchset reference the same robot run.

### 4. Dedup: write-on-success + Gerrit-side authoritative check

Gerrit's `webhooks` plugin delivers at-least-once and the review blows the ~5s socket
timeout, so the same event is re-delivered. Two-layer idempotency:

- A small **SQLite WAL** ledger on the box's data volume keyed by `(change_id,
  revision)`, recorded **write-on-success** â€” ONLY after a confirmed-successful vote.
  A failed vote leaves no row, so a retry re-attempts it.
- Before voting, the voter ALSO checks Gerrit for an existing non-zero `LLM-Review`
  vote on the current revision (`has_llm_review_vote`) â€” the **authoritative** guard,
  so a lost dedup row (fresh box) still cannot double-vote.

### 5. Single-flight critical section

A module-level dict of `asyncio.Lock` keyed by `(change_id, revision)` serializes a
webhook, its retries, and the backfill reconciler for the same patchset; the
dedup/Gerrit check INSIDE the lock makes the later arrivals a no-op skip.

### 6. ACK-fast / async processing

`POST /webhook` validates the inbound `?token=` (ADR-0014; constant-time compare),
enqueues the event, and returns **202 immediately**. A background worker drains the
queue and runs the reviewâ†’vote. Inline processing would blow Gerrit's webhook socket
timeout and trigger re-delivery against a still-running review.

### 7. Backfill reconciler

`reconcile.py` runs on startup and every `RECONCILE_INTERVAL_SECONDS` (default 300),
querying the `events-log` plugin (`GET /a/plugins/events-log/events/` â€” the TRAILING
SLASH is required) for patchsets whose current revision has no `LLM-Review` vote, and
re-invokes the SAME `review_and_vote` (sharing the lock + dedup). events-log is a
store, not an auto-replay; the reconciler is what recovers a dropped webhook.

### 7a. Manual `/rerun` recovery endpoint (force re-review, still fail-closed)

A transient outage (e.g. the LLM is briefly down) casts a fail-closed `-1` that then
sticks: the dedup ledger + the Gerrit existing-vote guard both make a re-delivery a
no-op skip, so the change stays unsubmittable with no automatic recovery. `POST /rerun`
is the operator escape hatch. Auth is the SAME `?token=` secret as `/webhook`
(constant-time compare). Given a `change` (id or number) it looks up the change's
CURRENT revision via `gerrit_client.get_change_event` (shaped like a `patchset-created`
event), enqueues it with a `_rebar_force` marker, and ACKs 202. The worker passes
`force=True` into `voter.review_and_vote`, which **bypasses both short-circuits** (the
dedup row AND the Gerrit existing-vote check) and re-reviews from scratch, overwriting
the stuck vote with a fresh verdict. It is **still fail-closed**: `force` can only
request a fresh review â€” the verdict is computed the same way, so a rerun can never cast
a PASS the reviewer did not produce. (A `force=False` call with a recorded vote still
skips, as before.)

### 7b. Reconciler: persisted cursor + events-log fallback

The reconciler no longer rescans the whole events-log every poll. It persists a
**cursor** â€” the newest events-log event time it has processed â€” in a small file
(`config.cursor_path`, default `<dedup dir>/reconcile_cursor`, on the same persistent
data volume as the dedup DB). Each pass fetches only events **since** that cursor via the
events-log REST time window (`GET /a/plugins/events-log/events/?t1=<yyyy-MM-dd HH:mm:ss>`,
UTC), reviews gap (vote-less) patchsets, then advances + persists the cursor (written
atomically: temp-file + replace) to the newest event seen. This makes the poller
**resumable** across restarts and cheap on a long-lived log. The cursor is purely an
optimization â€” **idempotency is still owned** by the per-`(change, revision)` dedup
ledger + the authoritative Gerrit vote-existence check, so a lost/reset cursor (or a
re-scan) can never double-vote.

**Fallback (degraded, fail-closed).** If events-log is absent, errors, or returns a
malformed (non-list) body, the reconciler logs a warning, emits a greppable
`RECONCILE_DEGRADED` marker (to journald, like `VOTER_ERROR`, so the host probe / alarm
can catch that backfill is degraded), does **NOT** advance the cursor, and casts **no**
vote â€” it relies on the live webhook path. A change a degraded backfill missed simply
stays vote-less = unsubmittable, never submittable-but-unreviewed.

### 8. Inbound vs outbound auth split

- **Inbound** (Gerrit â†’ receiver): the URL `?token=` secret + network ACL (ADR-0014).
  The `webhooks` plugin has no HMAC, so this is the only in-band credential.
- **Outbound** (receiver â†’ Gerrit REST): HTTP basic auth `BOT_USER:GERRIT_BOT_TOKEN`
  against the `/a/` namespace, reaching Gerrit over the compose network as
  `http://gerrit:8080` (NOT through public nginx).

### 9. Observability

Structured JSON logs throughout (`timestamp, change_id, revision_id, vote_value,
http_status, error`). A fail-closed failure emits a greppable `VOTER_ERROR` line; the
host observability probe greps the review-bot container's journald for it and
publishes `rebar/host:voter_errors`, watched by the `rebar-gerrit-voter-errors`
CloudWatch alarm (`infra/terraform/monitoring_s4b.tf`) â€” mirroring the S5
`replication_errors` pattern, so no AWS creds are needed inside the container.
The voter ALSO makes a **best-effort direct** `cloudwatch:PutMetricData` of the same
`rebar/host:voter_errors` metric via boto3 (instance role) from the `VOTER_ERROR` path,
wrapped so any `ImportError` / missing-credential / network failure is silently skipped.
In-container IMDS may be unreachable (the container IMDS hop limit can preclude it), so
**the journald â†’ host-probe path is the reliable fallback**; the direct publish is a
convenience when IMDS is reachable, never load-bearing.

## Consequences

- The receiver's structure (auth â†’ ACK â†’ queue â†’ worker â†’ adapter â†’ vote â†’ dedup) is
  fixed; b744-WS6 changes only `review_code`'s internals behind the `code_review_decision`
  seam and the severity/score mapping â€” no plumbing change.
- The gate is deterministic and fail-closed: one vote, one requirement, one voter, and
  every failure mode leaves the change unsubmittable rather than silently passing.
- **No new runtime dependency:** the Gerrit client is stdlib `urllib`/`subprocess`
  (the async voter runs those blocking calls via `asyncio.to_thread`), so the
  `reviewbot` extra stays `fastapi` + `uvicorn`.
