# Codebase Health Report — rebar — 2026-06-24

Audit by joe-janitor (8 parallel concern agents + temporal pass). Read-only.
Package: `src/rebar` — 180 Python files, 52.3K LOC, 376 test files.
History: full (not squashed), 529 commits over 16 days (2026-06-08 → 06-24), ~33 commits/day.

## Summary

- **Overall health: good for a 16-day AI-built greenfield.** No temporal decay signature
  (healthy 0.14–0.21 delete:add ratio, ~20% of recent commits do substantial restructuring,
  one clean strangler-fig `_engine`→`_commands` migration). Security posture is unusually
  strong (no `shell=True`, constant-time HMAC, hardened YAML loader, loopback-only editor
  with per-session token). The debt that exists is **structural** (size + cross-session
  duplication), not rot.
- **Top 3 risks:**
  1. **Phantom/under-declared dependency** — `jsonschema` is in the `dev` extra only and
     `referencing` is undeclared, yet both are imported by *core* (`schemas/__init__.py`) and
     the lean `run_workflow` path. A normal `pip install nava-rebar[agents]` install raises
     `ImportError` (or silently skips validation) on those paths. **[High]**
  2. **Cross-session duplication** — canonical-JSON+hash reimplemented 6+ times (already
     drifted on `ensure_ascii`, a hash-determinism axis), and 7+ competing retry/backoff
     strategies with hardcoded inconsistent attempt counts. Classic multi-agent fingerprint. **[High]**
  3. **The over-cap hotspot cluster** — the 7 most-frequently-changed live files are *all* at
     or over the 800-LOC soft cap (`_cli/__init__.py` 1262, `__init__.py` 1061, `config.py` 1048,
     `reads.py` 945, `mcp_server.py` 899, `reconcile.py` 1308 w/ a 759-LOC function, `applier.py`
     801). Complexity × change-frequency compounds here. **[High]**

## Findings (Critical → Low)

### [Critical] `reconcile_once()` is a 759-LOC monolithic orchestrator  (size, smells)
- **Citations:** `src/rebar/_engine/rebar_reconciler/reconcile.py:550-1308` (file 1308 LOC; the single function is ~759 LOC)
- **Justification:** Couples snapshot-load → diff → apply → persist → log plus corrupt-snapshot abort handling and OM→Mutation conversion in one body with 5-deep nested error handling (`:663-716`). Impossible to test a sub-phase or reuse the spine without invoking the whole orchestrator. This is the single worst unit in the tree, and it's the highest-touch reconciler file.
- **Tradeoffs:** The length reflects a deliberately consistent single-pass identity; splitting must preserve the idempotent contract.
- **Recommendation:** Extract `_load_snapshots()`, `_run_differs()`, `_apply_mutations()`, `_persist_and_log()` along the seams that already exist; keep `reconcile_once()` as a thin sequencer. Lift `_handle_corrupt_snapshot()` out of the spine.

### [High] Phantom / under-declared validation dependency (`jsonschema` dev-only, `referencing` undeclared)  (AI-code/security)
- **Citations:** declared `pyproject.toml:88` (under **`dev`**); imported by core/lean paths `src/rebar/schemas/__init__.py:280` (`from referencing import …`), `:294` (`from jsonschema import …`), `src/rebar/llm/workflow/interpreter.py:98`, `src/rebar/llm/findings.py:137,261`. **Verified.**
- **Justification:** CWE-1104. `referencing` (used directly) is in no extra; `jsonschema` is dev-only. `run_workflow` is documented as not needing `[agents]`, yet `interpreter.py:98` hard-imports `jsonschema.exceptions` with no guard → uncaught `ImportError` on a lean install running a workflow with an input contract. `findings.py` instead *swallows* the ImportError (`except Exception`) → output-schema validation silently doesn't run on `pip install nava-rebar[agents]` (which omits jsonschema).
- **Tradeoffs:** The engine deliberately keeps `dependencies = []`; adding a runtime dep cuts against that lean intent. But the code paths don't uniformly treat validation as optional.
- **Recommendation:** Add `jsonschema>=4.18` (pulls `referencing`) to a real runtime extra (`validation`, or fold into `agents`), and route every import through `rebar._optional.guard_import(..., extra=...)` so behavior is consistent (currently mixes silent-skip and hard-raise).

### [High] Canonical-JSON + content-hash reimplemented 6+ times, already drifted  (arch decay)
- **Citations:** `src/rebar/_store/canonical.py:42` (`ensure_ascii=False`), `src/rebar/_engine/rebar_reconciler/mutation.py:106` (`ensure_ascii=True`), `src/rebar/llm/workflow/schema.py:390-403`, `src/rebar/_engine/rebar_reconciler/conflict_resolver.py:215-219` (`default=str`), `src/rebar/llm/plan_review/orchestrator.py:688-689` (omits `separators`), `src/rebar/signing.py:187-200`. **`ensure_ascii` divergence verified.**
- **Justification:** Six independent "sorted-key compact JSON → sha256" sites; they've already diverged on encoding axes that change the digest for equal inputs. For content-addressing / signature / cross-clone-convergence primitives, encoding drift is exactly the bug class that breaks determinism. `_store/canonical.py` is the obvious home and is even named in `signing.py` docstrings, yet nobody imports it.
- **Tradeoffs:** Some variants may be load-bearing (`mutation.py`'s ASCII-only manifest); consolidation needs an audit of whether each difference is intentional.
- **Recommendation:** Promote one `canonical_json(doc)` + `content_hash(doc)` seam; make the sites import it. Express genuine variants as explicit named params, not copies.

### [High] 7+ competing retry/backoff implementations  (arch decay, code debt)
- **Citations:** `src/rebar/_store/push.py:25,122` (`_MAX_RETRIES=3`), `src/rebar/_engine/rebar_reconciler/acli_subprocess.py:234-287` (`max_retries=5`), `acli_rest.py:29-52` (`range(3)`), `_advisory_lock.py:106-126,432-469` (two budgets + two backoff fns), `_engine/jira-capability-probe.py:46-127` (`=6`), `llm/runner.py:298-317`.
- **Justification:** Seven distinct strategies, mutually-inconsistent attempt counts (3/3/5/5/6/8) and three backoff formulas — even two near-identical backoff fns inside one module. Transient-failure behavior is unpredictable and untunable from one place. Cross-session fingerprint.
- **Tradeoffs:** Lock-CAS vs HTTP-429 vs LLM-validation retries have genuinely different retryable predicates; one helper must be predicate+policy-parameterized to avoid an awkward abstraction.
- **Recommendation:** One `retry(fn, *, attempts, backoff, retry_on)` helper (or adopt `tenacity`); migrate the three HTTP/subprocess sites first; unify the two `_advisory_lock` backoff fns.

### [High] Over-cap hotspot cluster — 7 files at/over soft cap and top change-magnets  (size, temporal)
- **Citations:** `_cli/__init__.py:1262` (`_dispatch` 141-LOC, 30+ arms), `__init__.py:1061` (48 fns, read+write+LLM+workflow facade), `config.py:1048` (schema + override machinery), `_engine_support/reads.py:945` (CLI arms + state facades), `mcp_server.py:899` (`build_server` 601-LOC), `outbound_differ.py:1277`, `applier.py:801`.
- **Justification:** Temporal pass confirms these are the 7 highest-touch live `.py` files *and* all at/over the 800 cap — the "complex × frequently-changed" decay signature. Each carries 2+ unrelated responsibilities (the architecture-doc remedy table names the seams). They're stable now but every future feature re-enters them.
- **Tradeoffs:** All are grandfathered in `.github/module-size-allowlist.txt`, so CI won't force the split; doing it is voluntary churn against settled code. Per policy, split only along existing call-graph seams, never create <100-LOC files.
- **Recommendation:** Prioritize the data-driven splits the architecture doc already specifies: `_cli` routing table; `__init__` read/write API split; `config` schema vs override; `reads` `_cmd_*` arms vs `*_state` facades; `mcp_server` reads/writes/llm tool clusters.

### [High] Architecture-doc over-cap LOC table is stale on every row  (docs)
- **Citations:** `docs/architecture.md:194-204` vs actual `wc -l`: `_cli/__init__.py` doc 1100 → **1262**; `__init__.py` 1036 → **1061**; `reads.py` 930 → **945**; `mcp_server.py` 848 → **899**; `config.py` **doc 909 → 1048 (+139, worst)**.
- **Justification:** The table self-describes as "kept in lock-step" with the CI gate/allowlist, so an agent trusting it gets wrong size budgets. Every row undercounts; `config.py` is off by 139 lines.
- **Tradeoffs:** Exact LOC always lags slightly — but these are 100–160-line gaps, not rounding.
- **Recommendation:** Regenerate the LOC column + measurement date, or drop exact LOC and point at the allowlist as source of truth. (See also the freshly-fixed README CLI block — same drift class.)

### [High] CLI handlers + gates mix parsing, config I/O, LLM calls, and output shaping  (separation of concerns)
- **Citations:** `src/rebar/_cli/__init__.py:91-143,240-340` (`_review`/`_verify_completion`/`_review_plan`); duplicated gate logic in `src/rebar/_commands/transition.py:156-239` (`_completion_verification_gate`) and `src/rebar/_commands/claim.py:63-124` (`_plan_review_precheck`).
- **Justification:** The completion- and plan-review gate logic (config load → LLM verdict → audit log → fail-open-on-unreadable-config) is implemented twice, nearly identically, inlined in command handlers rather than a shared gates module — copy-paste-bug bait, and not reusable for a third gate.
- **Tradeoffs:** Inlining keeps each command self-contained and unit-testable; a gates module adds indirection.
- **Recommendation:** Extract `src/rebar/_commands/gates.py` with pure `check_completion_for_close()` / `check_plan_review_for_claim()`; CLI/commands call them and own only argv + output formatting.

### [High] Bare `except Exception` coverage gap (99 unguarded sites)  (code debt)
- **Citations:** e.g. `src/rebar/llm/plan_review/orchestrator.py:270-271,324-325` (`except Exception: pass`), `src/rebar/llm/completion.py:67`, `src/rebar/config.py:~500`; ~194 total, ~95 carry `# noqa: BLE001`, ~99 unguarded.
- **Justification:** Broad swallows in best-effort context-assembly paths hide real failures (e.g. tracker-dir permission errors) that should surface as warnings; operators get no signal. Concentrated in the hot `llm/plan_review` + reconciler paths.
- **Tradeoffs:** Many are intentional fail-open; refining to specific exception tuples is per-site judgement.
- **Recommendation:** `ruff check --select=BLE`; triage — add `# noqa: BLE001` + reason where intentional, else narrow to `(OSError, ValueError, json.JSONDecodeError, …)` and log at debug/warning.

### [High] `compute_outbound_mutations()` — 8 params incl. a mutable out-param  (smells)
- **Citations:** `src/rebar/_engine/rebar_reconciler/outbound_differ.py:1003-1275` (275-LOC body); out-param `absent_alive_fields` at `:1042-1052`.
- **Justification:** 8 params mixing always-needed inputs with optional behavioral overrides, plus an empty-dict-you-must-pass-and-inspect-after side-effect pattern. Confusing contract, hard to call correctly.
- **Tradeoffs:** A context object touches all call sites.
- **Recommendation:** `OutboundDiffConfig` dataclass for the optional flags; return `(mutations, absent_alive_fields)` instead of the out-param.

### [Medium] Output-schema validation fails open when validator deps absent  (AI-code/security)
- **Citations:** `src/rebar/llm/findings.py:140-142,259-262` (`except Exception: return data`).
- **Justification:** CWE-703. Combined with the phantom-dep finding, a standard `[agents]` install runs LLM ops with **no** structured-output enforcement — the `outputSchema` guarantee the docs advertise is effectively off in production installs, silently.
- **Recommendation:** Once jsonschema is in a runtime extra, make the absent-validator branch a hard error or loud one-time warning, not a silent pass.

### [Medium] Package dependency cycles held together by lazy imports  (arch decay)
- **Citations:** `graph↔_commands` (`graph/_links.py:130` vs `_commands/transition.py:28`, `composer.py:535,561`); `reducer↔_engine_support` (`reducer/_processors.py:225` imports `_engine_support.resolver` vs `_engine_support/reads.py:45-52`); `_cli↔_commands`.
- **Justification:** Three 2-package cycles survive only because back-edges are function-local lazy imports (`resolver.py:33` is even commented "lazy-import pair"). The `reducer` (pure replay layer) calling *up* into `_engine_support` for id resolution is a genuine layering inversion; a future top-level import turns it into an import-time crash.
- **Recommendation:** Extract the shared primitives to leaf modules (id/alias resolution → `rebar/_ids.py`; `_seam` → a module both `graph` and `_commands` depend on) so the back-edges disappear.

### [Medium] `_engine.py` module vs `_engine/` package name collision  (arch decay)
- **Citations:** `src/rebar/_engine.py:1` (asset locator / env builder) vs `src/rebar/_engine/` (the reconciler package it launches); `_native.py:12` imports `from rebar._engine import engine_dir` (resolves to the package).
- **Justification:** Same base name, unrelated roles, already entangled — brittle to refactor, misdirects edits.
- **Recommendation:** Rename the module to its role (`rebar/_engine_assets.py`) or fold its functions into `_engine/__init__.py`.

### [Medium] `_dispatch` CLI router — 30+ flat if/elif arms, inconsistent init  (smells, size)
- **Citations:** `src/rebar/_cli/__init__.py:1005-1145`.
- **Justification:** No routing table; each subcommand is a hand-inserted branch with inconsistent `ensure_initialized` handling and duplicated tracker resolution (`:1053,1098,1106,1120`).
- **Recommendation:** `DISPATCH_TABLE: dict[str, (handler, needs_init, init_only)]` + a shared `_get_tracker()`.

### [Medium] Deprecated `list_epics` still exported + wired into MCP  (code debt)

- **Justification:** CLAUDE.md calls `list_epics` deprecated in favor of composing `list_tickets(...)`, yet it's still a live public-API + MCP target. Ongoing maintenance + agent-confusion surface.

### [Medium] Library public-API wrappers duplicate identical error-mapping ceremony  (separation of concerns)

### [Medium] `rebar_reconciler/` directory: 42 files, 15.3K LOC, flat  (size)

## Looks bad but is fine (do not re-flag)

- **The five `acli*.py` files are ONE client deliberately split** (`acli.py` = `AcliClient`; `acli_rest.py`/`acli_graph.py` mixins; `acli_subprocess.py` transport; `acli_cli_ops.py` free-fn surface). Coherent module-size split, not competing Jira clients.
- **The two HMAC sites are NOT duplicated.** `llm/plan_review/attest.py:59,75` reuses `rebar.signing` verbatim; `_engine/rebar_reconciler/_attestation.py` is *GPG commit verification* (`git verify-commit`) — a different mechanism.
- **Three `config.py` namespaces are distinct concerns:** only the root `config.py` parses TOML; the reconciler `config.py` is pure constants; `llm/config.py` is env-var resolution. No duplicate loader.
- **`plan_review` finding shape differs from `llm/findings.py` by design** (binary sub-answers, no severity — documented at `llm/plan_review/passes.py:7-17`).
- **Security hardening is strong and was re-verified:** no `shell=True` anywhere; HMAC uses `hmac.compare_digest` and fails closed on empty key (`signing.py:256-280`); workflow YAML uses a hardened `SafeLoader` subclass; `random.*` is jitter-only (`# noqa: S311`); LLM file tools enforce realpath containment; editor binds loopback-only with per-session token. `hashlib.md5` (`reads.py:208`) is a non-security filesystem throttle-marker hash.
- **No temporal decay** — see Trends. The delete/refactor ratios are healthy for a greenfield build.
- **`lint_refs.py` (815, allowlisted)** should NOT be split yet — the natural `lint_contracts.py` seam is currently sub-100-LOC, below the policy floor. Defer until related stories (e050/c768) land.
